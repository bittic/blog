万有图书馆

万有图书馆

想象你站在一个无边无际的图书馆里，书架从你身边延伸出去，看不到尽头。你拿起一本书，里面是一串随机的文字；拿起下一本书，里面的内容仅仅和上一本书差了一个字；拿起再下一本书，里面的内容又只变化了一个字。

在这个图书馆里，任意两本相邻的书，都只变化了一个字。书架无边无际，变化无穷无尽。这，就是万有图书馆。

可以想象，绝大多数书里，只有无意义的混乱的字符排列。但只要你在万有图书馆里徜徉足够长的时间，你一定会找到一本书，里面的内容是一个有意义的单词，一句话，甚至一本莎士比亚全集。

实际上，万有图书馆里，记录着全人类有史以来说过的所有话、写过的所有书、以及所有尚未出现的文字。它们隐藏在无边无际的书架上，等待着有人走过发现，也许永远离发现差了一点点 – 但它们必已存在、必将在某个时候出现。

应用与数据分离

每个用户设定自己的密语 secword，并由此可推导出任意多个子密钥对（分层确定性钱包算法 HD）。

某用户 creator 通过某应用 app，想要存入一本新书 book：

  1. 客户端对 book 计算出 bookhash，和当前用户地址一起，发给万有书库；
  2. 万有书库检查该 bookhash 是否已存在。如果存在，返回警告；如果不存在，返回该用户的交易次数 n
  3. 客户端求出当前用户的第 n 个密钥对：HD(secword, n) => 密钥对 (pubkey, seckey)
  4. 客户端用自己的公钥 creator_pubkey 对 bookhash 进行加密，得到加密数据 creator_cipher
  5. 客户端用代理平台方的公钥 proxy_pubkey 给 bookhash 进行加密，得到加密数据 proxy_cipher
    1. 这样，平台方可以在主人不参与的情况下，反解出数据给访问者
    2. 如果不加密，直接存原 bookhash 就是 NFT 了，大家都可以读，但所有权是主人的
    3. 这一步可选。如果不存原 bookhash 也不让平台解密，就是不允许访问者了。
  6. 客户端对 { creator_cipher, proxy_cipher, pubkey, fee, 平台 pubkey } 用 creator_seckey 进行签名，发往万有书库。
  7. 万有书库验证签名后，从用户账户里扣除“存入费”、转给应用账户 app、把 { cipher, 用户 pubkey, fee, 平台 pubkey } 存入万有书库。这一系列操作形成一次交易，记入万有书库区块链
  8. 交易哈希 txhash 作为图书地址返回给用户

某用户 reader 想要读取某 txhash 里的内容：

  1. 客户端查询万有书库，获得该 txhash 对应的 bookhash
  2. 万有书库有权利在收到费用后，把 bookhash 给 reader
  3. 主人有权利转账该 bookhash 给下一任主人

用公私钥还是哈希做加密？
自己存放，还是存在 ipfs ？
存放完整数据，还是仅仅存放哈希？
每个节点存储一个区块，不同节点可以存储同一份资料，不同持有人。使用者之间转账 连接到其他区块链上，例如 eth，或者实现自己的内置区块链。
每个节点上：一个大区块存储数据，一个区块链用于发币

用户输入内容
内容处理成哈希，原始数据（或经过加密，或拆分）存入 ipfs 或后台数据库
哈希存入区块链

主权，溯源，自带收费

主旨：发表链上作品，自动订阅、销售、转让

  • 创作者发表作品
    • 作品的内容存入后台服务器，或者 ipfs
    • 作品的哈希记入后台的本机区块链（用区块链做数据结构，但不需要基于 p2p 网络，只在后台本机存放）
  • 围绕作品的用户操作
    • 用户付费访问作品：作品默认显示为哈希链接，用户点击后被要求付费，付费后即可查看作品内容
    • 求购/出售、转让一个作品：作品的所有权从创作者开始进行流转。每次被用户付费访问时，访问费在原创者-持有人-平台三方之间分成。
  • 支付流程
    • 最简化方案：应用内部代币（对应于一个以太坊代币，但实际上是应用内部转账的）。
      • 优点：减免用户使用 eth/usdt 的门槛、减免以太坊转账损耗。
      • 缺点：该代币没有交易所定价
    • 用户使用 eth/usdt 付费，应用需要对接 metamask 或者 etherscan.io

比特币，以太坊，NFT 的实现
https://blog.csdn.net/liudaoqiang_tj/article/details/81545002

  • 比特币

    • 一对密钥 - 一个地址 - 多个 UTXO，每个里面含有一定数量的未花费的比特币
    • 每一笔交易 - 多个输入（UTXO+签名）+多个输出（UTXO）
    • 不存在账户，只有钱包，里面管理了若干密钥及其地址
    • UTXO 独立可并行，用挖矿解决双花攻击
    • UTXO 的设计正可以看成是借鉴了现金的思路：我们可能在这个口袋里装点现金，在那个柜子角落里放点现金，在这种情况下不存在一个账户，你放在各处的现金加起来就是你所有的钱。

  • 以太坊

    • 一对密钥 - 一个地址 - 一个账户，里面有余额
    • 以太坊：私钥账户，合约账户
    • 账户不可并行，每个账户有一个从 0 开始随着交易次数而递增的 nonce, 用于解决重放攻击
